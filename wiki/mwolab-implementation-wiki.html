MwoLab 구현 위키

기준일 2026-07-23 · 현재 저장소 구현 기준

이 문서는 MwoLab의 현재 구조, 계산 기준, UI 동작과 수정 시 주의사항을 한곳에 모은 내부 구현 가이드다.

문서 운용 원칙
이 위키는 MwoLab 프로젝트와 분리된 단일 HTML 파일이다. 앱에는 링크를 추가하지 않는다. 자동으로 갱신하지 않으며, 사용자가 명시적으로 “위키 갱신”을 요청한 경우에만 현재 구현을 다시 확인해 수정한다.

현재 상태

MwoLab은 MechWarrior Online의 로컬 게임 데이터를 정적 JSON으로 변환해 사용하는 브라우저 기반 멕 피팅·정보·비교 도구다. 별도 프레임워크나 서버 API 없이 public 폴더만으로 배포할 수 있다.

구조와 데이터 흐름

HTML이 화면 골격을 제공하고, 하나의 전역 상태 객체를 중심으로 app.js가 데이터 로딩·계산·렌더링·이벤트를 처리한다. MWO 로드아웃 문자열 변환만 mwo-codec.js에 분리되어 있다.

MWO 설치 파일 → extract_mwo_data.py → public/data/*.json → init() → state → 계산 함수 → 화면 렌더링
  1. init()이 index.json과 멕·장비·로드아웃·옵니포드 JSON을 읽는다.
  2. 읽은 데이터는 state에 저장되고, 검색·쿼크·하드포인트용 파생 인덱스는 브라우저 메모리 캐시에 만든다.
  3. 멕 선택 시 기본 로드아웃 또는 저장된 빌드를 currentBuild로 구성한다.
  4. calculateBuild()가 중량, 슬롯, 아머, 장비, 탄약, 엔진, 발열과 경고를 계산한다.
  5. 각 탭의 렌더 함수가 동일한 원본 데이터와 계산 기준을 사용해 화면을 갱신한다. 보이지 않는 주 탭은 미리 렌더링하지 않는다.

추출된 숫자는 게임 데이터 갱신 전까지 사실상 정적이다. 반복 계산이 큰 값은 JSON에 다시 저장하지 않고 브라우저 메모리에서 캐시한다.

  • 정보 탭은 아머, 스트럭처, 종합 내구도, 엔진 범위, 기동성, 쿼크를 보여 준다.
  • 비교 탭은 선택 멕 중 하나를 기준으로 삼아 상승·하락을 표시하며 고정 헤더를 사용한다.
  • 정보·비교·통계에서 쿼크 적용 기준이 서로 어긋나지 않아야 한다.

브라우저 저장

  • 기본 저장 키: 멕별 저장 빌드.
  • 이름 있는 로컬 빌드: mwolab:local-builds:v1.
  • 쿼크 값 표시 방식: mwolab:quirk-value-display의 final, quirk, all.
  • 서버 계정이나 원격 저장소는 사용하지 않는다.

피팅 규칙

컴포넌트 순서

Head → Left Arm → Left Torso → Center Torso → Right Torso → Right Arm → Left Leg → Right Leg 순서를 유지한다. 후방 토르소 아머는 독립 장비 카드가 아니라 아머 계산에만 포함한다.

슬롯과 하드포인트

  • 사용 슬롯 = 컴포넌트 내부 고정 항목 수 + 장착 아이템의 stats.slots 합계다.
  • 슬롯 한도는 컴포넌트의 slots를 사용한다.
  • 무기 장착 수용량은 하드포인트의 weapon_slots를 사용한다. 원본 MDF의 Slots 속성은 무기 개수가 아니다.
  • weapon_slots 매핑이 실제로 없는 경우에만 1을 폴백으로 사용한다.
  • 옵니멕은 현재 장착한 옵니포드 하드포인트를 최우선으로 사용한다.
  • 하드포인트 ID는 선행 0을 제거해 정규화한 뒤 비교한다.

장비 필터와 장착

  • 에너지·미사일·발리스틱·AMS는 해당 유효 하드포인트가 있을 때만 목록에 표시한다.
  • ECM, 점프젯, MASC는 멕 능력과 톤수 조건을 모두 만족해야 한다.
  • C.A.S.E., Active Probe, Target Computer처럼 원본 데이터에 장착 수량이나 허용 부위가 지정된 장비는 그 제한을 검증한다.
  • 드래그 앤 드롭과 클릭 자동 장착은 동일한 dropValidation 규칙을 거친다.
  • 장착 후 중량 초과, 슬롯 초과, 하드포인트 부재, 엔진 범위, 장비 수량 제한을 계산해 경고한다.

업그레이드와 MWO 코드

  • 아머, 스트럭처, 히트싱크, 아르테미스 업그레이드를 빌드 상태로 관리한다.
  • 아르테미스 적용/해제 시 표시 이름만 바꾸지 않고 무기와 탄약 ID를 양방향으로 교체한다.
  • 표준 → 아르테미스 → 표준 왕복 후 원래 ID가 복원되어야 한다.
  • alwaysHasArtemis 무기는 선택형 변환 대상이 아니다.
  • mwo-codec.js는 6비트 문자 형식의 직렬화만 담당하고, 실제 멕·장비 검증은 app.js에서 한다.

쿼크 규칙

  • 쿼크 이름과 무기 검색 키는 대소문자를 구분하지 않도록 정규화한다.
  • 요약 값은 일반 쿼크 + 무기 계열 쿼크 + 가장 잘 맞는 무기 전용 쿼크를 합친다.
  • 쿨다운·발열·지속시간·산포처럼 감소가 이득인 multiplier는 음수로 저장되므로 Math.max(0, -value)로 요약한다.
  • 사거리·탄속·열 방출·ROF처럼 증가가 이득인 multiplier는 Math.max(0, value)로 요약한다.
  • 무기 전용 쿼크는 통계 접미사를 제거한 이름을 장비의 이름·표시 이름·별칭과 매칭한다.
  • 무기 계열은 개별 이름 하드코딩보다 장비 데이터의 hardpoint_type 또는 stats.type을 우선한다.

요약 카드

  • 쿨다운: MAX, ENERGY, MISSILE, BALLISTIC.
  • 발열: MAX, ENERGY, MISSILE, BALLISTIC, HEAT DISSIPATION.
  • 사거리·탄속: MAX, ENERGY, MISSILE, BALLISTIC.
  • 듀레이션/ROF: 레이저 듀레이션과 가장 높은 ROF를 5% / 3% rof 형태로 한 행에 표시한다.
  • 산포: 에너지 항목이 없지만 시각 정렬을 위해 빈 위치를 유지한다.
  • 내구도: 부위별이 아니라 MAX(아머+스트럭처), 아머, 스트럭처, 크리티컬 방지의 총합만 표시한다.
  • 특수: 현재 ECM, 점프젯, NARC 지속시간을 인식한다.

쿼크 서머리는 QUIRKS 목록 안이 아니라 독립 정보 카드다.

값 색상

기본값과 쿼크 기여도를 함께 표시하는 경우 최종값은 초록색, 쿼크 기여도는 노란색이다. 기본값, 괄호, 연산자는 주변 통계의 기본 색을 유지한다.

HPD와 MASC 툴팁

  • HPD는 열 1점당 피해량이므로 damage / heat로 계산한다.
  • MASC 최종 기동성은 쿼크가 반영된 현재 값에 MASC 부스트를 곱한다: 쿼크 최종값 × (1 + MASC boost).
  • MASC 부스트가 0인 항목은 부스트 행과 최종값 행을 모두 표시하지 않는다.

에너지 레이저 예외

레이저 전용 쿼크는 레이저 무기에, 에너지 쿼크는 모든 에너지 무기에 적용한다. PPC도 에너지 쿼크의 영향을 받는다. 이름에서 구분되는 경우 표준 레이저 쿼크를 ER 또는 펄스 변형에 섞지 않는다.

무기 계산 규칙

피해와 연속 무기

  • 직격 피해는 기본적으로 damage × numFiring × 발사체 수다.
  • 탄환형 펠릿 무기는 numPerShot을 발사체 수에 반영한다.
  • 스플래시 총피해는 직격 + 양옆 부위의 스플래시를 합산한다.
  • 머신건·로터리 AC는 직격 피해 × ROF를 초당 피해로 표시하고 ROF 쿼크를 반영한다.
  • 레이저·화염방사기처럼 원본 피해가 초당 값인 연속 무기는 ROF를 다시 곱하지 않는다.
  • 레일건 커패시터 등 항상 적용되는 무기 모듈 보너스는 동일 진영의 장비 데이터에서 찾아 피해와 발열에 합산한다.

사거리와 특수 통계

  • 툴팁 사거리 순서는 MIN RANGE, OPTIMAL RANGE, MAX RANGE, VELOCITY다.
  • 사거리 쿼크는 MIN RANGE를 늘리지 않는다.
  • ATM은 60~349m, 350~649m, 650~1100m의 피해 구간을 사용하며 사거리 쿼크는 구간 상한과 최대 사거리에 반영한다.
  • 산포, 크리티컬 확률·피해, 대상 발열, 차지 시간은 데이터가 있을 때 별도 통계로 표시한다.
  • Ultra AC는 잼 지속시간, 잼 확률, 기대 DPM을 표시하며 관련 쿼크를 반영한다.

탄약

  • 탄약량은 장착 탄약 레코드의 stats.numShots를 사용하며 반톤과 1톤을 따로 존중한다.
  • 활성 탄약 타입은 선택형 아르테미스 적용 시 artemisAmmoType, 그 외에는 ammoType이다.
  • 동일한 정규화 탄약 타입을 쓰는 무기들은 탄약 풀을 공유한다.
  • 동시 발사 가능 횟수는 매칭되는 탄약통의 합계를 묶인 무기들의 총 ammoPerShot으로 나눈다.
  • 탄약에도 적용되는 전용 무기 쿼크를 장비 필터와 비교 통계에 일관되게 반영한다.

고스트 힛

데이터와 계산 규칙

  • 외부 웹사이트의 표를 수치 원본으로 사용하지 않는다. 무기별 기본 발열, 페널티, 발생 시점, 그룹은 equipment.json의 heat, heatpenalty, minheatpenaltylevel, heatPenaltyID에서 읽는다.
  • 같은 heatPenaltyID의 무기를 동시에 발사해 무기별 발생 시점을 넘으면 고스트 힛 후보를 계산한다. 개별 그룹은 같은 무기끼리만 영향을 준다.
  • 같은 그룹 안에서도 AC/20과 Clan AC/20처럼 발생 시점이 다를 수 있고, AC/20과 Ultra AC/20처럼 추가 발열도 다를 수 있다.
  • 동시에 발생한 후보 중 가장 높은 추가 발열 하나만 기본 발열 합계에 더한다.
  • 고정 단계 상승 곡선은 코드에 두되, 무기별 수치와 그룹은 JSON에서 가져온다.

무장 정보

  • 그룹별 표에는 무기마다 각 동시 발사 수에서 생기는 추가 발열만 표시한다.
  • 그룹명은 IS MRM/SRM GROUP, IS ROTARY AC GROUP, IS LRM/THUNDERBOLD GROUP 등을 사용한다.
  • 공유 그룹이 없는 무기는 개별 그룹 - 개별 그룹으로 서로 영향 없음으로 표시한다.
  • 설명은 최고 추가 발열 적용 규칙과 AC/20 계열, AC/10 ×2 + Ultra AC/10 ×1 예시로 요약한다.

멕랩 표시

  • 현재 피팅에서 고스트 힛 발생 조건을 만족하면 슬롯 영역 왼쪽 위에 경고 아이콘을 오버레이한다. 아이콘 표시 여부는 슬롯 위치를 바꾸지 않는다.
  • 발생 그룹에 참여하는 이동식·고정 무장 슬롯에는 빨간 외곽선을 표시한다.
  • 경고 아이콘 툴팁은 발생 그룹마다 한 줄씩 무기 구성과 최고 추가 발열을 표시하며 내용에 맞춰 폭을 정한다.
  • 발생 슬롯의 HEAT 툴팁은 최종 결과 모드에서 최종수치 + 고스트 힛, 쿼크 수치 모드에서 최종수치(쿼크수치) + 고스트 힛, 모든 수치 모드에서 최종수치(원본 ± 쿼크) + 고스트 힛 순서로 표시한다.
  • 고스트 힛 수치는 빨간색 굵은 글씨로 구분한다. 멕랩의 ALPHA, HEAT, DPS 기본 수치에는 합산하지 않는다.

사격 시뮬레이션

현재 빌드에 장착된 무기를 가져와 브라우저 애니메이션 프레임으로 사격, 쿨다운, 발열, 냉각, 잼과 피해 누적을 계산한다.

  • 사격 그룹 할당과 누름 상태를 관리하며 그룹별 시각 상태를 갱신한다.
  • 거리, 이동 상태, 맵 온도, 스플래시 적용, 과열 시 종료를 시나리오 조건으로 사용한다.
  • 일반 발사, 연속 피해, 번(burn), 연발 대기 탄환을 각각 처리한다.
  • 쿨다운·발열·사거리·ROF·무기 전용 쿼크와 타겟 컴퓨터 보너스를 적용한다.
  • 같은 시점에 발사된 무기를 고스트 힛 그룹별로 묶고 그룹에서 가장 높은 추가 발열을 시뮬레이션 열에 적용한다.
  • 경과 시간, 총 피해, DPS, 발열은 무기 발사 컨트롤 바로 위에 함께 둔다.
  • 시뮬레이션 UI는 게임 배치에 가깝게 하되 시야와 즉시 판독성을 우선한다.

데이터 추출

의도적인 게임 데이터 갱신은 반드시 tools/extract_mwo_data.py를 사용한다.

원본용도
Game/GameData.pak멕 목록, 장비, 로드아웃, 아르테미스 원본 속성
Game/Localized/English_xml.pak표시 이름과 현지화 텍스트
Game/mechs/*.pak차체·옵니포드 상세 정의와 하드포인트 WeaponSlot 수

전체 추출

python tools/extract_mwo_data.py --game-dir "F:\Game\Steam\steamapps\common\MechWarrior Online" --out public\data

하드포인트만 갱신

python tools/extract_mwo_data.py --game-dir "F:\Game\Steam\steamapps\common\MechWarrior Online" --out public\data --hardpoints-only

전체 추출은 설치된 게임의 모든 데이터를 다시 읽기 때문에 요청 범위 밖의 밸런스 값이나 숫자 형식도 바뀔 수 있다. 하드포인트 수정만 필요하면 --hardpoints-only로 기존 장비·로드아웃 데이터를 보존한다.

추출 데이터 보존 원칙

  • 원본 속성은 덮어쓰지 않고 필요한 파생 필드를 추가한다.
  • 무기에는 ammoType, artemisAmmoType, alwaysHasArtemis를 보존한다.
  • 탄약에는 type, numShots를 보존한다.
  • 표준/아르테미스 무기와 탄약 대응 레코드가 실제로 존재하는지 검증한다. 누락 시 이름 치환으로 조용히 폴백하지 않는다.
  • 파생 요약과 캐시를 public/data/*.json에 영구 저장하지 않는다.

변경 검증

수정 범위에 비례해 아래 항목을 확인한다. 사용자가 요청하지 않으면 로컬 미리보기 서버를 시작·탐색·검증하지 않는다.

  • JavaScript와 Python 구문 검사.
  • git diff --check로 공백·패치 오류 검사.
  • 생성 JSON이 바뀐 경우 변경 파일 범위와 예상 밖 데이터 변경 확인.
  • 정보·비교·통계·멕랩이 동일한 계산 기준을 쓰는지 확인.

  • 기존 사용자의 관련 없는 변경은 유지하고 필요한 부분만 좁게 수정.
